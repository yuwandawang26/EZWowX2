## 1. 文档概述

### 1.1 项目背景

EZWowX2 是一个《魔兽世界》游戏辅助系统，通过像素化技术将游戏内数据转换为外部程序可读的格式。该项目采用多语言混合架构，包括游戏内Lua插件、像素解析服务（.NET/Python）和决策执行引擎（Python）。

### 1.2 文档目的

本文档定义了EZWowX2项目的功能需求、非功能需求、用户场景及验收标准，为项目开发、测试和交付提供依据。

### 1.3 术语表

| 术语 | 定义 |
|-----|------|
| 像素块 (Pixel Block) | 游戏内将战斗信息编码为特定颜色的 8x8 像素区域 |
| 节点 (Node) | 像素块在网格中的坐标位置 |
| 遮挡检测 | 验证游戏窗口是否被遮挡导致数据失真 |
| 决策上下文 | 用于决策判断的游戏状态快照 |
| 节点提取器 | 从像素帧中提取游戏数据的组件 |
| 循环引擎 | 负责调度、决策调用、输入发送的核心组件 |

## 2. 项目组成模块

### 2.1 模块架构总览

```
EZWowX2/
├── EZAddonX2/                    # 游戏内基础插件（简易API）
├── EZPixelAddonX2/              # 游戏内像素编码插件（高级协议）
├── EZPixelDumperX2.NET/         # .NET 像素捕获与解析服务
├── EZPixelDumperX2/              # Python 像素捕获与解析服务
├── EZBridgeX2/                  # 桥接服务与图标数据库
├── EZDriverX2/                   # 决策执行引擎
├── EZPixelRotationX2/            # 像素旋转决策
└── EZAssistedX2/                # 辅助操作模块
```

### 2.2 各模块职责

| 模块 | 技术栈 | 职责 |
|-----|-------|------|
| EZAddonX2 | Lua | 游戏内基础UI框架和数据展示 |
| EZPixelAddonX2 | Lua | 像素编码协议实现，将游戏数据编码为像素 |
| EZPixelDumperX2.NET | C#/.NET | 像素捕获、解析、HTTP API服务 |
| EZPixelDumperX2 | Python | Python版像素捕获与解析 |
| EZBridgeX2 | Python | 桥接服务、图标数据库管理 |
| EZDriverX2 | Python | 决策循环、配置管理、动作执行 |
| EZPixelRotationX2 | Python | 决策逻辑实现 |
| EZAssistedX2 | Lua + .NET | 辅助操作集成 |

## 3. 功能需求

### 3.1 像素编码与解码系统

#### 3.1.1 像素协议定义

**FR-001: 像素块结构定义**

- 每个像素块大小为 8x8 像素
- 像素块内部分为多个区域：
  - `full`: 完整 8x8 区域
  - `middle`: 中心 6x6 区域（用于哈希计算）
  - `inner`: 内层 4x4 区域
  - `footnote`: 右下角 2x2 区域（用于图标类型标识）
  - `sub_node`: 四个 2x2 子区域

**FR-002: 颜色编码规范**

| 颜色类型 | RGB值 | 用途 |
|---------|-------|------|
| MAGIC | 60,100,220 | 魔法类型 |
| CURSE | 100,0,120 | 诅咒类型 |
| DISEASE | 160,120,60 | 疾病类型 |
| POISON | 154,205,50 | 中毒类型 |
| ENRAGE | 230,120,20 | 激怒类型 |
| BLEED | 80,0,20 | 流血类型 |
| PLAYER_DEBUFF | 255,60,60 | 玩家减益 |
| PLAYER_BUFF | 80,220,120 | 玩家增益 |
| PLAYER_SPELL | 64,158,210 | 玩家施法 |
| ENEMY_SPELL_INTERRUPTIBLE | 255,255,60 | 可打断敌方施法 |
| ENEMY_SPELL_NOT_INTERRUPTIBLE | 200,0,0 | 不可打断敌方施法 |
| ENEMY_DEBUFF | 105,105,210 | 敌方减益 |

**FR-003: 节点坐标映射**

| 节点坐标 | 用途 |
|---------|------|
| (0,0)-(1,1) | 左上角锚点 |
| (50,16)-(51,17) | 右下角锚点 |
| (1,1) | 参考色点1 |
| (50,16) | 参考色点2 |
| (2,2)-(37,2) | 玩家技能序列 |
| (2,5)-(33,5) | 玩家增益序列 |
| (2,8)-(9,8) | 玩家减益序列 |
| (10,8)-(25,8) | 目标减益序列 |
| (26,8)-(33,8) | 焦点减益序列 |
| (38,2)-(45,4) | 玩家状态 |
| (38,6)-(45,7) | 目标状态 |
| (38,8)-(45,9) | 焦点状态 |
| (1,11)-(47,15) | 队伍信息 |

#### 3.1.2 像素解析功能

**FR-004: 模板匹配与定位**

- 支持通过模板图像定位游戏数据区域
- 使用锚点坐标验证定位准确性
- 支持多显示器环境

**FR-005: 图标识别**

- 通过xxhash3_64计算图标哈希值
- 支持精确哈希匹配
- 支持余弦相似度回退匹配（阈值0.995）
- 图标数据库存储：SQLite

**FR-006: 数据提取**

- 从像素帧中提取以下数据：
  - 玩家/目标/焦点单位状态
  - 法术序列（冷却、充能、可用性）
  - 光环序列（buff/debuff）
  - 队伍成员状态
  - 信号状态
  - 专精信息

### 3.2 数据传输系统

#### 3.2.1 HTTP API服务

**FR-007: REST API接口**

- 默认端口: 65131
- 基础URL: http://127.0.0.1:65131/
- 响应格式: JSON
- 支持GET请求

### 3.3 决策执行系统

#### 3.3.1 循环调度

**FR-009: 主循环控制**

- 支持启动/停止后台线程执行循环
- tick间隔计算: `base_interval = 1/fps`，支持抖动(jitter)
- 失败路径使用短退避: `interval/5`
- 策略函数异常必须被捕获并记录，循环不可中断

**FR-010: 配置系统**

- 支持以下配置项：
  - `fps`: 1-30，默认15
  - `interval_jitter`: 0.0-0.5，默认0.2
  - `spell_queue_window`: 0.1-0.5，默认0.2
- 配置写入权限控制（仅GUI可写）

#### 3.3.2 数据适配

**FR-011: AttrDict数据转换**

- 字典转点号访问对象
- 递归转换嵌套字典与列表内字典
- 访问不存在键时返回NoneObject占位对象

**FR-012: NoneObject语义**

- 任意属性访问返回自身
- 布尔值为假
- 可迭代为空
- 长度为0
- 与None视为等价

#### 3.3.3 动作执行

**FR-013: 键盘输入发送**

- 目标窗口句柄校验
- 组合键发送规则：按下顺序、逆序抬起
- 按压间隔: 10ms ± 5ms
- 支持键范围: CTRL/SHIFT/ALT + NUMPAD0-9 + F1-F12

**FR-014: 动作映射**

- 映射键拼接规则: `target + spell`
- 映射不存在时记录告警，不中断循环

### 3.4 遮挡检测系统

**FR-015: 锚点验证**

- 节点(1,16)必须为黑色
- 节点(50,1)必须为黑色
- 节点(1,1)必须为纯色（参考色）
- 节点(50,16)必须为纯色（参考色）
- 节点(1,1)和(50,16)颜色必须匹配
- 节点(51,4)必须为非纯色（数据区）

**FR-016: 遮挡处理**

- 检测到遮挡时返回带error的JSON
- 不输出脏数据

## 4. 非功能需求

### 4.1 性能需求

| 指标 | 目标值 | 说明 |
|-----|-------|------|
| 帧处理延迟 P99 | <50ms | 从捕获到数据输出 |
| 决策循环周期标准差 | <10% | 保持稳定帧率 |
| API响应时间 | <10ms | 不包含网络延迟 |
| 内存增长率 | <1%/min | 防止内存泄漏 |

### 4.2 可靠性需求

| 指标 | 目标值 | 说明 |
|-----|-------|------|
| 连续失败帧容忍 | 3帧 | 超过后触发警告 |
| 异常恢复能力 | 自动重试 | 可恢复错误自动重试 |
| 进程稳定性 | 零崩溃 | 长时间运行稳定 |

### 4.3 兼容性需求

- 操作系统: Windows 10/11
- 游戏版本: WoW 12.0+
- Python版本: 3.12+
- .NET版本: .NET 8.0+

### 4.4 安全需求

- 仅本地通信，不涉及网络传输
- 不修改游戏内存
- 不注入游戏进程

## 5. 用户场景

### 5.1 场景一：戒律牧自动治疗

**前置条件**:
1. 玩家已加载EZPixelAddonX2插件
2. 已启动EZPixelDumperX2服务
3. 已启动EZDriverX2/PixelRotation并选择戒律牧配置

**执行流程**:
1. PixelAddon在游戏内显示像素编码
2. Dumper捕获屏幕并解析像素数据
3. Rotation通过HTTP API获取数据
4. 根据策略判断并执行动作
5. 通过Win32 API发送按键指令

**验收标准**:
- [ ] 成功获取玩家/目标/队伍数据
- [ ] 治疗阈值触发时自动施放技能
- [ ] 异常情况下不崩溃

### 5.2 场景二：多目标战斗

**前置条件**:
同5.1

**执行流程**:
1. 获取多个目标状态
2. 优先级排序
3. 按顺序执行动作

**验收标准**:
- [ ] 正确识别多个目标
- [ ] 优先级判断正确
- [ ] 动作执行顺序正确

### 5.3 场景三：游戏窗口遮挡恢复

**前置条件**:
Dumper正在运行

**执行流程**:
1. 用户切换到其他窗口
2. Dumper检测到遮挡（锚点验证失败）
3. 返回error状态的JSON
4. Rotation处理error并等待恢复
5. 用户切回游戏
6. Dumper检测到恢复
7. 恢复正常数据输出

**验收标准**:
- [ ] 遮挡时正确返回error
- [ ] 恢复后自动继续
- [ ] Rotation不因遮挡而崩溃

## 7. 假设与约束

### 7.1 假设

- 游戏窗口始终可见（除遮挡检测场景外）
- 用户具有管理员权限（用于Win32 API调用）
- 显示器分辨率为1080P或更高

### 7.2 约束

- 不支持Linux/macOS（仅Windows）
- 不支持全屏窗口捕获（仅窗口模式）
- 不支持游戏内直接API调用
